# Conceptual documentation in `docs/model/`

Long-form docs that explain _why_ and _how_. Code is authoritative for
_what_; don't mirror field types or APIs here.

Each doc has a `Created: YYYY-MM-DD` header and covers one concept.
Avoid status hedges ("currently," "for now," "later") and project-state
mentions ("Phase X done") — those rot. Route project state to ADRs or
inline TODOs instead.
